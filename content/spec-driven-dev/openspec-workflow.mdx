# OpenSpec 六阶段工作流详解

## 概述

OpenSpec 定义了一个六阶段工作流，覆盖从需求探索到部署上线的完整生命周期。每个阶段都有明确的输入、输出和 AI 协作模式。

```
Discover → Define → Design → Develop → Validate → Deploy
   ↓          ↓        ↓         ↓         ↓          ↓
  探索       定义     设计      开发      验证       部署
```

让我们深入每个阶段，了解实际操作流程和最佳实践。

## 第一阶段：Discover（发现）

### 目标
明确「为什么做」和「做什么」，建立对问题的共同理解。

### 输入
- 用户反馈或业务需求
- 竞品分析
- 技术调研
- 现有系统的限制和痛点

### AI 协作模式
在这一阶段，AI 扮演「分析师」角色，帮助你：

- **整理和归类需求**：将散乱的用户反馈整理为结构化的需求列表
- **生成用户故事**：从需求描述自动生成标准格式的用户故事
- **竞品分析辅助**：基于公开信息快速生成竞品对比分析

### 实践示例

```markdown
# Discover 阶段 Prompt 模板

我正在规划一个「团队知识库」功能。请帮我完成以下分析：

1. 目标用户画像：谁会用这个功能？他们的核心痛点是什么？
2. 核心场景：列出 5 个最重要的使用场景
3. 成功指标：如何衡量这个功能是否成功？
4. 风险识别：列出可能的失败模式和应对策略
5. 竞品参考：主流知识库产品（Notion、Confluence、飞书文档）在这个领域的做法

请以结构化方式输出，每个部分给出具体建议。
```

### 产出物
- `discover/user-stories.md`：用户故事列表
- `discover/problem-statement.md`：问题陈述文档
- `discover/research-notes.md`：调研笔记

### 实践技巧

- **不要跳过这一步**：很多团队急于进入实现阶段，但前期探索不充分是项目失败的首要原因
- **让 AI 帮你问「愚蠢的问题」**：有时我们会忽略显而易见的边缘情况，让 AI 列出 10 个可能被忽略的问题
- **保持发现文档简洁**：这不是论文，2-3 页的要点总结比 20 页的详细分析更有价值

## 第二阶段：Define（定义）

### 目标
将模糊的需求转化为精确的规范，定义「做成什么样才算完成」。

### 输入
- 第一阶段产出的用户故事和问题陈述
- 技术约束和限制
- 项目时间线和优先级

### AI 协作模式
AI 在此阶段充当「技术写作助手」：

- **生成功能规范草稿**：根据用户故事生成详细的功能描述
- **识别边界条件**：列出所有需要明确处理的边缘情况
- **定义验收标准**：生成可测试的 Given-When-Then 格式验收标准

### 实践示例

```markdown
# Define 阶段 Prompt 模板

基于以下用户故事，生成详细的功能规范：

用户故事：作为团队成员，我希望能够创建和编辑富文本文档，以便记录和分享团队知识。

请输出：
1. 功能概述（一段话）
2. 详细功能描述（包括所有子功能）
3. UI/UX 要求
4. 数据模型初稿
5. API 端点列表
6. 验收标准（使用 Given-When-Then 格式）
7. 非功能需求（性能、安全、可访问性）
```

### 产出物
- `define/feature-spec.md`：功能规范文档
- `define/acceptance-criteria.md`：验收标准
- `define/data-model.md`：数据模型定义

### 实践技巧

- **使用 Given-When-Then 模板**：
  ```
  Given 用户已登录且拥有编辑权限
  When 用户点击「新建文档」按钮
  Then 系统创建一个空白文档并跳转到编辑页面
  And 文档自动保存到用户的私有空间
  ```
- **区分 Must Have 和 Nice to Have**：使用 MoSCoW 方法（Must/Should/Could/Won't）标注优先级
- **与利益相关者确认**：在进入设计阶段前，确保关键方对规范达成一致

## 第三阶段：Design（设计）

### 目标
制定技术方案，决定「怎么做」——选择架构模式、技术选型、接口设计。

### 输入
- 第二阶段产出的功能规范
- 项目现有技术栈
- 非功能需求（性能、安全、可扩展性）

### AI 协作模式
AI 作为「技术架构顾问」：

- **生成架构方案**：根据规范提出多种技术方案
- **方案对比分析**：对比不同方案的优劣
- **生成接口定义**：产出 OpenAPI 规范、GraphQL Schema 等
- **识别技术风险**：分析潜在的技术债务和安全风险

### 实践示例

```markdown
# Design 阶段 Prompt 模板

基于以下功能规范，请提供技术设计方案：

功能：团队知识库 - 支持富文本编辑、版本历史、协作编辑

请包含：
1. 架构概述（附架构图描述）
2. 技术栈推荐及理由（对比至少 2 个方案）
3. 数据流图描述
4. 核心 API 接口设计（使用 OpenAPI 格式）
5. 数据库 Schema 设计
6. 关键技术挑战及解决方案
7. 安全设计要点
```

### 产出物
- `design/architecture.md`：架构设计文档
- `design/api-spec.yaml`：API 规范（OpenAPI/GraphQL）
- `design/database-schema.sql`：数据库 Schema
- `design/component-tree.md`：组件树（前端项目）

### 实践技巧

- **让 AI 给出多个方案并对比**：不要满足于 AI 给出的第一个方案。使用以下 Prompt：
  > 「针对这个设计问题，请给出 3 种不同的技术方案，并对比它们的优劣（复杂度、性能、可维护性、团队学习成本）」

- **架构决策记录（ADR）**：为每个重要决策创建 ADR，记录背景、决策内容和后果：

  ```markdown
  # ADR-001: 选择 Zustand 作为状态管理方案

  ## 背景
  项目需要全局状态管理，需要在 Redux、Zustand、Jotai 之间选择。

  ## 决策
  选择 Zustand，原因如下：
  1. 极简 API，学习成本低
  2. TypeScript 支持优秀
  3. Bundle 体积小（< 1KB）
  4. 无需 Provider 包裹

  ## 后果
  - 正面：开发效率高，代码简洁
  - 负面：大型应用状态组织不如 Redux Toolkit 规范
  ```

## 第四阶段：Develop（开发）

### 目标
将设计文档转化为可运行的代码。

### 输入
- 第三阶段产出的设计文档
- 功能规范和验收标准
- 项目编码规范

### AI 协作模式
AI 作为「结对编程伙伴」：

- **根据规范生成代码**：将设计文档作为上下文，生成实现代码
- **增量开发**：按照规范拆解的任务逐步实现
- **代码审查**：对照规范检查实现是否正确

### 实践示例

```markdown
# Develop 阶段 Prompt 模板

请根据以下规范和设计文档，实现「文档列表页面」组件：

## 功能规范
[引用 define/feature-spec.md 中的相关部分]

## 设计文档
[引用 design/component-tree.md 中的组件设计]

## 技术约束
- 使用 React 18 + TypeScript
- 样式使用 TailwindCSS
- 使用 React Query 进行数据获取
- 遵循项目的 ESLint 规则

请生成：
1. 组件代码（包括 types、hooks、主组件）
2. 单元测试
3. Storybook story
```

### 产出物
- 实现代码
- 单元测试
- 集成测试
- Storybook stories（如适用）

### 实践技巧

- **分步实现，逐层验证**：不要一次性生成所有代码。按以下顺序进行：
  1. 先生成类型定义和接口
  2. 再生成数据层（API 调用、状态管理）
  3. 最后生成 UI 组件
  4. 每步完成后进行验证

- **将规范文件作为 AI 上下文**：在 Prompt 中引用规范文件的具体部分，而不是依赖 AI 的记忆

- **使用「规范约束」模式**：
  ```
  请严格遵循以下规范实现功能，不要偏离：
  - API 响应格式必须符合 [api-spec.yaml] 中的定义
  - 错误处理必须使用 [error-handling.md] 中定义的模式
  - 组件必须符合 [component-standards.md] 中的命名和结构规范
  ```

## 第五阶段：Validate（验证）

### 目标
确保实现符合规范，功能正确，质量达标。

### 输入
- 第四阶段的实现代码
- 第二阶段的验收标准
- 第三阶段的设计文档

### AI 协作模式
AI 作为「质量保障工程师」：

- **对照规范检查实现**：逐条验证实现是否满足验收标准
- **生成测试用例**：根据规范自动生成测试用例
- **代码审查**：检查代码质量、安全性、性能
- **文档一致性检查**：确保代码与文档一致

### 实践示例

```markdown
# Validate 阶段 Prompt 模板

请对照以下规范，审查已实现的代码：

## 验收标准
[引用 define/acceptance-criteria.md]

## 检查清单
1. 所有验收标准是否已满足？
2. 是否存在未处理的边缘情况？
3. 错误处理是否完善？
4. 性能是否符合非功能需求？
5. 是否存在安全漏洞？
6. API 响应格式是否与规范一致？

对于每个未满足的项，请给出具体的修复建议。
```

### 产出物
- `validate/test-report.md`：测试报告
- `validate/review-notes.md`：审查记录
- `validate/compliance-check.md`：规范符合性检查

### 实践技巧

- **自动化规范符合性检查**：尽可能编写脚本自动检查 API 响应格式、数据模型等是否符合规范
- **分层验证**：
  1. 单元测试：验证单个函数/组件
  2. 集成测试：验证模块间交互
  3. E2E 测试：验证用户流程
  4. 规范符合性：对照规范逐条检查

- **让 AI 从规范生成测试**：
  ```
  基于以下验收标准，生成完整的测试用例：

  验收标准：用户登录功能
  Given 用户输入正确的邮箱和密码
  When 用户点击「登录」按钮
  Then 系统返回 JWT token
  And 用户被重定向到首页

  请生成：
  1. 单元测试（验证登录逻辑）
  2. 集成测试（验证 API 端点）
  3. E2E 测试（使用 Playwright）
  ```

## 第六阶段：Deploy（部署）

### 目标
将经过验证的代码部署到生产环境，并建立监控和反馈机制。

### 输入
- 第五阶段通过验证的代码
- 部署配置
- 监控和告警需求

### AI 协作模式
AI 作为「运维顾问」：

- **生成部署配置**：Docker、Kubernetes、CI/CD 配置
- **编写部署文档**：部署步骤和回滚方案
- **生成监控配置**：关键指标和告警规则

### 产出物
- `deploy/deployment-guide.md`：部署指南
- `deploy/rollback-plan.md`：回滚方案
- `deploy/monitoring-config.md`：监控配置
- CI/CD 配置文件

### 实践技巧

- **自动化一切**：部署流程应该完全自动化，人工操作是事故的主要来源
- **建立反馈循环**：部署后的监控数据应反馈到 Discover 阶段，形成持续改进的闭环

```
Discover → Define → Design → Develop → Validate → Deploy
    ↑                                                   ↓
    └─────────── 反馈循环 ←────── 监控数据 ←─────────────┘
```

## 工作流实战：一个完整示例

让我们通过一个真实案例来展示六阶段工作流如何运作。

### 场景：为电商平台添加「商品收藏」功能

**Discover 阶段输出**：
```markdown
# 问题陈述：商品收藏功能

用户痛点：用户浏览商品时，想保存感兴趣的商品以便后续比较和购买，
但当前只能通过浏览器书签或复制链接的方式保存，体验很差。

核心场景：
1. 在商品列表/详情页收藏商品
2. 在个人中心查看收藏列表
3. 从收藏列表直接加入购物车
4. 批量管理收藏（删除、分类）
5. 商品降价/补货时收到通知
```

**Define 阶段输出**：
```markdown
# 验收标准

AC-1: 收藏商品
Given 用户已登录且在商品详情页
When 用户点击「收藏」按钮
Then 按钮状态变为「已收藏」
And 商品被添加到用户的收藏列表

AC-2: 取消收藏
Given 用户已登录且商品已在收藏列表
When 用户点击「取消收藏」
Then 商品从收藏列表中移除
And 按钮状态恢复为「收藏」

AC-3: 未登录收藏
Given 用户未登录
When 用户点击「收藏」按钮
Then 弹出登录提示
And 登录成功后自动完成收藏操作
```

**Design 阶段输出**：
```typescript
// API 接口设计
POST   /api/favorites          // 添加收藏
DELETE /api/favorites/:id      // 取消收藏
GET    /api/favorites          // 获取收藏列表（支持分页）
GET    /api/favorites/check    // 检查是否已收藏

// 数据模型
interface Favorite {
  id: string;
  userId: string;
  productId: string;
  createdAt: Date;
  tags: string[];          // 用户自定义标签
  priceAtFavorite: number; // 收藏时的价格
}
```

**Develop 阶段**：AI 根据规范和设计生成完整的前后端代码。

**Validate 阶段**：AI 生成测试用例并自动验证。

**Deploy 阶段**：通过 CI/CD 自动部署，配置收藏转化率监控。

## 常见问题与解答

### Q: 六个阶段是否太繁琐？敏捷开发强调快速迭代。

A: 阶段数量不等于流程缓慢。对于小功能，六个阶段可能每个只需 10-30 分钟。关键在于每个阶段都有明确的产出物，确保质量不因速度而妥协。

### Q: 如何处理规范变更？

A: 规范变更遵循「提案 → 审查 → 批准 → 更新」的流程。变更被批准后，同步更新所有相关阶段文档，确保一致性。

### Q: AI 生成的规范质量如何？

A: AI 可以生成 80% 质量的规范初稿，但最后的 20%——特别是业务逻辑和边缘情况——需要人类判断。最佳实践是 AI 生成初稿 + 人工精炼。

## 结语

OpenSpec 的六阶段工作流提供了一条从需求到上线的完整路径。它不是僵化的流程，而是一个灵活的框架——你可以根据项目规模和复杂度调整每个阶段的深度。

关键要点：
1. **每个阶段都有明确产出物**，不跳步不偷懒
2. **AI 在每个阶段都能提供价值**，但人类判断不可替代
3. **规范是活的文档**，随项目演化而更新
4. **从简开始，逐步深化**，不要一开始就试图做到完美
