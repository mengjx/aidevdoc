# 开发者必知的 Prompt Engineering 模板与技巧

## 为什么 Prompt Engineering 对开发者至关重要？

2025 年，编写 Prompt 已经成为开发者的核心技能之一。好的 Prompt 和差的 Prompt 之间的差距，可能意味着 10 分钟 vs 2 小时的工作效率差异。

但 Prompt Engineering 不是玄学——它是一套可以通过学习和练习掌握的系统化技能。本文将提供可复制的模板和经过验证的技巧。

## 五大 Prompt 模式

### 模式 1：规范优先（Spec-First）

**核心理念**：在让 AI 写代码之前，先让它生成一份详细的技术规范。

**为什么有效**：
- 避免 AI 在没有充分理解需求的情况下直接生成代码
- 让 AI 先「思考」再「行动」
- 产出的规范可以作为后续开发的参考文档

**基础模板**：

```
我想实现以下功能：[功能描述]

请先生成一份技术规范，包括：
1. 功能概述
2. 输入输出定义
3. 数据结构和接口
4. 关键算法/逻辑描述
5. 边界条件和异常处理
6. 测试场景

规范生成后，我会审查并确认，然后你再根据规范生成代码。
```

**进阶模板（含约束条件）**：

```
## 任务背景
我们正在开发一个 [项目类型]，技术栈是 [技术栈]。
现有架构采用 [架构模式]。

## 需求
[详细需求描述]

## 约束条件
- 必须兼容现有的 [模块/API]
- 性能要求：[具体要求]
- 代码风格必须符合项目规范：[编码规范]
- 不要引入新的依赖，除非绝对必要

## 输出要求
请按以下结构输出：

### 1. 需求分析
分析需求的核心挑战和关键技术决策点。

### 2. 技术方案对比
给出 2-3 种实现方案，对比优劣。

### 3. 推荐方案详细设计
针对推荐方案，给出：
- 数据结构定义
- 接口签名
- 核心流程伪代码
- 错误处理策略

### 4. 实现注意事项
列出实现时需要注意的陷阱和边缘情况。

请先输出以上规范，不要直接生成实现代码。
```

### 模式 2：链式思考调试（Chain-of-Thought for Debugging）

**核心理念**：让 AI 逐步分析问题，而不是直接猜测答案。

**基础模板**：

```
我遇到了以下 bug：

## Bug 描述
[详细描述问题现象]

## 复现步骤
1. [步骤1]
2. [步骤2]
3. [观察到的异常]

## 相关代码
```[语言]
[粘贴相关代码]
```

## 错误信息
[粘贴完整的错误堆栈]

请按以下步骤分析：
1. 解读错误信息，定位最可能的错误来源
2. 分析代码中可能导致该错误的地方
3. 提出 2-3 种可能的修复方案
4. 推荐最佳方案并给出修改后的代码
```

**进阶模板（根本原因分析）**：

```
## Bug 报告

### 现象
[详细描述]

### 环境信息
- 运行环境：[OS, 运行时版本]
- 依赖版本：[关键依赖及版本]

### 相关日志
```
[粘贴日志]
```

### 最近变更
- [变更1]
- [变更2]

## 分析要求

请进行系统性的根本原因分析：

### 步骤 1：问题定位
基于错误信息和日志，确定问题发生在哪个层级（网络、应用、数据库等）。

### 步骤 2：假设生成
列出 3 个最可能的原因假设，按可能性排序。

### 步骤 3：假设验证
对每个假设，说明如何验证（添加日志、断点、测试用例）。

### 步骤 4：根本原因
给出你认为最可能的根本原因。

### 步骤 5：修复方案
给出修复代码和验证方法。

### 步骤 6：预防措施
如何防止类似问题再次发生？
```

### 模式 3：角色扮演（Role-Based Prompting）

**核心理念**：为 AI 指定专业角色，激发特定领域的知识和思维模式。

**代码审查专家模板**：

```
请以资深 [技术栈] 架构师的身份，审查以下代码。

审查维度：
1. 代码质量：命名、结构、可读性
2. 性能：潜在的瓶颈和优化建议
3. 安全性：OWASP Top 10 风险检查
4. 可维护性：设计模式和代码组织
5. 错误处理：异常处理的完整性
6. 测试覆盖：缺失的测试场景

```[语言]
[粘贴代码]
```

请以结构化格式输出审查结果，每个问题标注严重程度（🔴 严重 / 🟡 警告 / 🟢 建议）。
```

**性能优化专家模板**：

```
请以性能优化专家的身份，分析以下代码的性能问题：

```[语言]
[粘贴代码]
```

分析要求：
1. 时间复杂度分析（标注最坏情况）
2. 空间复杂度分析
3. 识别性能瓶颈（具体到行）
4. 提出优化方案（包括优化前后的对比数据预估）
5. 如果涉及数据库查询，分析查询计划和索引建议
6. 如果涉及网络请求，分析批处理和缓存策略

优化方案请按「投入产出比」排序，优先推荐低成本高收益的优化。
```

### 模式 4：方案对比（Give Me 3 Approaches）

**核心理念**：强制 AI 探索多种方案，避免单一思路的限制。

**基础模板**：

```
我需要实现 [功能描述]。

请给出 3 种不同的实现方案，每种方案包含：
- 核心思路（一句话）
- 优点（3 条）
- 缺点（3 条）
- 适用场景
- 核心代码骨架

最后给出一个对比总结表格，并推荐最佳方案。
```

**进阶模板（含决策矩阵）**：

```
## 需求
[详细需求]

## 技术约束
- 语言/框架：[约束]
- 性能要求：[约束]
- 兼容性要求：[约束]
- 团队经验：[约束]

## 要求

请提出 3 种实现方案，覆盖不同的设计理念（如：简洁优先、性能优先、可扩展优先）。

对每个方案，输出：
1. 核心设计思路
2. 架构草图（文字描述或 Mermaid 图）
3. 关键代码示例
4. 优缺点分析

最后，基于以下维度进行评分（1-5 分）并汇总：
- 实现复杂度（越低越好）
- 性能表现
- 可维护性
- 可扩展性
- 团队学习成本（越低越好）
- 与现有系统的兼容性

根据评分矩阵给出最终推荐。
```

### 模式 5：增量精炼（Iterative Refinement）

**核心理念**：不要期望一次 Prompt 得到完美结果。通过多轮对话逐步精炼。

**工作流模板**：

```
第 1 轮（探索）：
「帮我理解 [概念/问题]，用简单的话解释。」

第 2 轮（规划）：
「基于以上理解，帮我制定一个实现计划。」

第 3 轮（实现）：
「根据计划，实现第一步。」

第 4 轮（审查）：
「审查刚才的代码，提出改进建议。」

第 5 轮（优化）：
「根据建议优化代码。」

第 6 轮（测试）：
「为这段代码编写完整的测试用例。」

第 7 轮（文档）：
「为这个功能编写使用文档。」
```

## 上下文窗口管理技巧

随着 AI 模型上下文窗口的增大（Claude 支持 200K tokens），有效管理上下文变得更重要。

### 技巧 1：分层上下文策略

不要把所有信息都塞进 Prompt。采用分层策略：

```
第一层（必须）：当前任务的直接相关信息
第二层（重要）：相关的项目规范和设计文档
第三层（参考）：相关的已有代码
第四层（补充）：项目整体背景
```

**实践**：
```markdown
## 当前任务（第一层）
修改用户认证中间件，增加 JWT 刷新逻辑。

## 相关规范（第二层）
- API 响应格式必须符合 [api-spec.yaml]
- 错误码定义见 [error-codes.md]

## 关键代码（第三层）
```typescript
// 当前认证中间件
[粘贴核心代码]
```

## 项目背景（第四层）
项目采用 NestJS + Prisma，JWT 认证基于 @nestjs/jwt。
```

### 技巧 2：使用引用而非复制

对于大型规范文件，使用文件引用而非全文复制：

```
请参考以下文件中的相关内容：
- 项目规范：CLAUDE.md（已索引）
- API 设计：/specs/api-spec.yaml
- 编码规范：/docs/coding-standards.md

请先确认你已理解这些文件的内容，然后开始任务。
```

### 技巧 3：摘要化长对话

当对话历史过长时，让 AI 生成摘要以保持上下文：

```
请将我们之前的讨论总结为以下格式：

1. 项目背景（3 句话）
2. 已完成的修改（列表）
3. 当前待解决的问题
4. 关键技术决策
5. 下一步计划

这个总结将用于新会话的上下文初始化。
```

### 技巧 4：上下文文件（CLAUDE.md / .cursorrules）

创建项目级的上下文文件，让 AI 在任何会话中都能获取关键信息：

```markdown
# CLAUDE.md / .cursorrules

## 项目概述
这是一个 [项目描述]，使用 [技术栈]。

## 架构决策
- 架构模式：[如 Feature-Sliced Design]
- 状态管理：[如 Zustand]
- API 风格：[如 RESTful / GraphQL]

## 编码规范
- 组件：React Functional Components + TypeScript
- 样式：TailwindCSS，使用 utility-first 方式
- 命名：PascalCase 组件，camelCase 函数和变量
- 文件组织：每个 feature 一个目录

## 禁止事项
- 不要使用 any 类型
- 不要引入新的依赖而不告知
- 不要修改 .eslintrc 或 tsconfig 而不讨论

## 常用命令
- 开发：`npm run dev`
- 测试：`npm test`
- 构建：`npm run build`
```

## 常见 Prompt 错误与纠正

### 错误 1：需求过于模糊

**❌ 错误**：
```
帮我写一个用户管理功能。
```

**✅ 正确**：
```
帮我实现用户管理功能，包括：
1. 用户列表页面（支持分页、搜索、按角色筛选）
2. 用户详情页面
3. 用户创建/编辑表单
4. 用户删除（软删除）

技术栈：React + TypeScript + Ant Design
API 已就绪，接口文档见：[链接]
请先给出实现计划，不要直接写代码。
```

### 错误 2：缺少约束条件

**❌ 错误**：
```
优化这段代码的性能。
```

**✅ 正确**：
```
优化以下代码的性能：

```javascript
[代码]
```

优化重点：
1. 减少不必要的渲染（使用 React.memo、useMemo、useCallback）
2. 优化循环和数据结构选择
3. 如果涉及 API 调用，添加防抖/节流

约束：
- 不改变外部接口
- 保持代码可读性
- 不要过度优化
```

### 错误 3：一次性要求太多

**❌ 错误**：
```
帮我设计、实现、测试并部署一个完整的电商系统。
```

**✅ 正确**：
```
我想构建一个电商系统。让我们从第一步开始：

先帮我设计系统的整体架构和数据模型。

技术栈倾向：
- 前端：Next.js
- 后端：Node.js
- 数据库：PostgreSQL

请先给出架构概览和核心数据模型。
```

## 高级技巧

### 技巧：让 AI 自检

```
在生成代码后，请进行自我检查：
1. 是否满足所有需求？
2. 是否存在潜在 bug？
3. 是否有更好的实现方式？
4. 错误处理是否完整？
5. 是否有安全隐患？

如有问题，请自动修正后重新输出。
```

### 技巧：渐进式复杂度

```
让我们从最简单的情况开始，逐步增加复杂度：

第 1 步：实现最基本的功能（不考虑边界情况）
第 2 步：添加错误处理
第 3 步：添加加载状态
第 4 步：处理边界情况
第 5 步：添加性能优化

每步完成后我会确认，然后进行下一步。
```

### 技巧：反向 Prompt

```
请告诉我，为了让你更好地完成这个任务，我应该提供哪些额外信息？
```

这个技巧让 AI 告诉你它需要什么，而不是你猜测它需要什么。

## 结语

Prompt Engineering 不是要记住复杂的咒语，而是掌握一套系统化的思维框架：

1. **明确需求**：让 AI 先理解，再行动
2. **结构化输出**：告诉 AI 你期望的输出格式
3. **分步推进**：复杂任务拆解为小步骤
4. **约束边界**：明确技术栈、规范、限制
5. **迭代精炼**：通过多轮对话逼近最优解

将这些模板保存到你的笔记工具中，在实际开发中反复使用和调整。优秀的 Prompt Engineering 能力，是 2025 年开发者效率差距的关键因素。
