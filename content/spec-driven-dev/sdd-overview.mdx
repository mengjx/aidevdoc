# 规范驱动开发（SDD）完全指南：从概念到实战

## 从 Vibe Coding 到 Spec Coding：范式转变

2024 年底，Andrej Karpathy 提出了 **Vibe Coding** 的概念——开发者通过自然语言描述需求，让 AI 直接生成代码，不再逐行编写。这种模式在早期非常令人兴奋：只需几句话，AI 就能生成一个完整的 Web 应用。

然而，Vibe Coding 的局限性很快暴露出来：

- **不可预测性**：相同的 prompt 在不同时刻可能产生截然不同的代码
- **缺乏一致性**：AI 生成的代码风格、架构决策在多次对话中难以保持统一
- **维护噩梦**：当项目规模增长，没有结构化文档的 AI 生成代码变得难以理解和修改
- **上下文丢失**：当对话历史过长或开启新会话时，AI 失去了对项目全貌的理解

这些问题催生了一个新的开发范式：**Spec-Driven Development（SDD，规范驱动开发）**。

## 什么是规范驱动开发？

规范驱动开发是一种将 **结构化规范文档** 置于开发流程核心的方法论。它的核心思想是：

> **先写规范，再让 AI 根据规范生成代码。**

这里的「规范」不是传统意义上的 PRD（产品需求文档），而是一组 **可执行、可验证、结构化的技术文档**，通常包括：

1. **功能规范**：详细描述系统行为的结构化文档
2. **技术约束**：技术栈、架构模式、编码标准
3. **验收标准**：可测试的条件，用于验证实现是否符合规范
4. **变更提案**：对规范的修改遵循类似 RFC 的流程

### SDD 与 Vibe Coding 的对比

| 维度 | Vibe Coding | Spec-Driven Development |
|------|------------|------------------------|
| 开发起点 | 自然语言闲聊 | 结构化规范文档 |
| 代码一致性 | 低，随对话变化 | 高，基于规范约束 |
| 可复现性 | 难以复现相同结果 | 规范驱动，结果可预期 |
| 团队协作 | 个人对话式 | 规范即协作文档 |
| 长期维护 | 困难 | 规范即文档，易于理解 |
| 上下文管理 | 依赖对话历史 | 规范文件提供持久上下文 |

## SDD 的核心价值

### 1. 上下文永续性

在传统 AI 辅助开发中，每次新对话都意味着上下文丢失。SDD 通过将关键信息固化为规范文件，确保 AI 在任何会话中都能获取完整上下文。这意味着：

- 新加入的开发者可以快速了解项目全貌
- AI 模型切换或会话重启不会丢失关键信息
- 规范文件本身成为项目的「单一事实来源」（Single Source of Truth）

### 2. 意图与实现分离

SDD 强制开发者先明确「做什么」（What），再让 AI 处理「怎么做」（How）。这种分离带来：

- 更清晰的架构决策
- 更少的返工（因为前期规范已经考虑到边界情况）
- 更好的可测试性（验收标准在编码前就定义好了）

### 3. 渐进式精确化

规范可以从高层抽象逐步细化到具体实现细节，这天然适配 AI 辅助开发的工作方式：

```
高层愿景 → 功能规范 → 技术设计 → 实现任务 → 验收测试
```

每一步都产出可审查的文档，每一步都可以由 AI 辅助完成。

## 三大 SDD 框架对比

目前主流的 SDD 框架有三个：**OpenSpec**、**GitHub Spec Kit** 和 **Kiro**。它们在设计理念和工作流上有显著差异。

### OpenSpec

**OpenSpec** 是一个开放的规范驱动开发框架，强调 **渐进式规范演化** 和 **AI 原生协作**。

**工作流（六阶段）**：
1. **Discover（发现）**：探索需求，收集上下文
2. **Define（定义）**：明确功能边界和验收标准
3. **Design（设计）**：制定技术方案和架构决策
4. **Develop（开发）**：根据设计文档生成代码
5. **Validate（验证）**：对照规范进行测试和审查
6. **Deploy（部署）**：发布并收集反馈

**核心优势**：
- 完全开源，社区驱动
- 与主流 AI 编码工具（Claude Code、Cursor）深度集成
- 灵活的规范格式，支持 Markdown 和结构化数据
- 强调「规范即文档」，减少文档维护成本

**适用场景**：从个人项目到企业级应用都适用，尤其适合需要长期维护的项目。

### GitHub Spec Kit

**GitHub Spec Kit** 是 GitHub 推出的规范驱动开发工具包，深度集成 GitHub 生态（Issues、Projects、Actions）。

**工作流**：
1. 创建 Spec Issue 描述功能需求
2. AI 根据 Issue 生成详细 Spec 文档
3. 团队 Review 并批准 Spec
4. AI 根据 Spec 自动拆解为子任务
5. 逐任务实现，自动关联 Issue
6. CI 验证通过后自动关闭 Issue

**核心优势**：
- 与 GitHub 生态无缝集成
- 内置任务管理和进度追踪
- 适合开源项目的协作模式
- GitHub Actions 原生支持 CI/CD 验证

**局限性**：
- 强依赖 GitHub 平台
- 规范格式相对固定，定制性有限
- 对新项目的初始化支持不如 OpenSpec 灵活

### Kiro

**Kiro** 是 Anthropic 推出的规范驱动开发环境，强调 **交互式规范探索** 和 **可视化开发**。

**工作流**：
1. 通过对话探索需求和设计空间
2. AI 自动生成交互式规范原型
3. 在可视化环境中验证规范
4. 一键从规范生成完整项目
5. 持续对话迭代

**核心优势**：
- 交互式、可视化的规范编辑体验
- 与 Claude 深度集成，理解能力更强
- 快速原型验证
- 适合探索性开发

**局限性**：
- 相对封闭的生态
- 大型项目的规范管理能力尚待验证
- 生成的代码与规范的对应关系不够透明

### 三框架综合对比

| 维度 | OpenSpec | GitHub Spec Kit | Kiro |
|------|----------|----------------|------|
| 开放性 | 完全开源 | GitHub 生态内 | 相对封闭 |
| 规范格式 | Markdown + 结构化 | Issue + Markdown | 交互式 UI |
| AI 集成 | 多工具支持 | GitHub Copilot | Claude 专属 |
| 任务管理 | 规范内嵌 | GitHub Issues | 内置管理 |
| 学习曲线 | 中等 | 低（GitHub 用户友好） | 低（交互式引导） |
| 企业就绪度 | 高 | 中 | 待验证 |
| 社区活跃度 | 快速增长 | GitHub 官方支持 | 新兴 |

## 如何选择适合你的 SDD 框架？

### 选择 OpenSpec，如果你：
- 需要跨平台、跨工具的灵活性
- 项目需要长期维护，且希望保持技术文档的独立性
- 团队使用多种 AI 编码工具（Cursor、Claude Code、Copilot 等）
- 重视社区驱动和开源生态

### 选择 GitHub Spec Kit，如果你：
- 项目已经深度使用 GitHub 生态
- 团队习惯基于 Issue 的工作流
- 需要 GitHub Actions 的 CI/CD 集成
- 参与或维护开源项目

### 选择 Kiro，如果你：
- 偏好可视化和交互式开发体验
- 正在做探索性项目或原型验证
- 团队主要使用 Claude 系列模型
- 不需要过于复杂的项目管理功能

## 从 Vibe Coding 迁移到 SDD 的实践路径

如果你目前在使用 Vibe Coding，以下是一个循序渐进的迁移路径：

### 第一步：引入项目上下文文件

在项目根目录创建 `CONTEXT.md` 或 `CLAUDE.md`，包含：

```markdown
# 项目上下文

## 技术栈
- 前端: React 18 + TypeScript + TailwindCSS
- 后端: Node.js + Express + Prisma
- 数据库: PostgreSQL

## 架构决策
- 采用 Feature-Sliced Design 架构模式
- API 使用 RESTful 风格，遵循 OpenAPI 3.0 规范
- 状态管理使用 Zustand

## 编码规范
- 使用 ESLint + Prettier，配置见 .eslintrc.js
- 组件命名遵循 PascalCase
- 函数命名遵循 camelCase
```

### 第二步：为每个功能编写功能规范

```markdown
# 功能规范：用户认证模块

## 概述
实现基于 JWT 的用户认证系统，支持邮箱注册和登录。

## 功能列表
- [ ] 用户注册（邮箱 + 密码）
- [ ] 邮箱验证
- [ ] 登录 / 登出
- [ ] 密码重置
- [ ] Session 管理

## API 设计
POST /api/auth/register - 用户注册
POST /api/auth/login - 用户登录
POST /api/auth/verify-email - 邮箱验证
POST /api/auth/reset-password - 密码重置

## 验收标准
- 密码使用 bcrypt 加密存储
- JWT token 有效期 24 小时
- 支持 refresh token 机制
- 注册后发送验证邮件
```

### 第三步：采用规范优先的开发流程

1. **新功能开发前**：先编写功能规范
2. **规范审查**：团队 Review 规范文档
3. **AI 辅助实现**：将规范提供给 AI 工具进行开发
4. **对照验证**：基于验收标准进行测试
5. **规范更新**：实现完成后更新规范，保持文档与代码一致

### 第四步：引入正式的 SDD 框架

当团队适应了规范优先的工作方式后，引入 OpenSpec 或 GitHub Spec Kit 进行更系统化的规范管理。

## SDD 的挑战与应对

### 挑战 1：规范编写成本

**应对**：将规范视为投资而非成本。一份好的规范可以减少 30-50% 的返工时间。同时利用 AI 辅助生成规范初稿，人工进行审查和精炼。

### 挑战 2：规范与代码的同步

**应对**：建立「规范即文档」的文化，将规范更新作为 PR 审查的检查项。可以使用自动化工具检查规范与实现的一致性。

### 挑战 3：过度规范

**应对**：采用「恰到好处」的规范粒度。不是每个函数都需要规范，重点是模块边界、API 契约和核心业务逻辑。

## 结语：SDD 是 AI 开发的下一个里程碑

Vibe Coding 让我们看到了 AI 辅助开发的潜力，但它的随意性限制了其在严肃项目中的应用。SDD 提供了一条中间道路——既保留 AI 的高效性，又引入结构化规范来保证质量和可维护性。

随着 AI 模型能力的持续提升，SDD 的价值会更加凸显：当 AI 能够完美执行规范时，规范的质量将直接决定最终产出的质量。**编写优秀的规范，正在成为 AI 时代开发者最重要的技能之一。**
